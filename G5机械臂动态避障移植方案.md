## 结论

G5 的 `/scan -> /cmd_vel` 脚本只能测底盘动态避障响应，不能直接完成机械臂避障。机械臂避障需要把障碍物转换成 MoveIt PlanningScene 里的 `CollisionObject`，再让 MoveIt 在规划时避开。

## 最小移植闭环

第一阶段只做“手动注入动态障碍盒”，不接入视觉，不改 pick/place：

1. 在机械臂工作空间中选一个障碍盒位置。
2. 通过 `/apply_planning_scene` 把障碍盒加入 MoveIt PlanningScene。
3. 让现有 MoveIt 规划一次会经过该区域的动作。
4. 观察规划是否绕开、失败、或仍穿过障碍。
5. 移除障碍盒，重复对比。

这个闭环验证的是：项目现有 MoveIt 配置是否真的消费 PlanningScene 障碍物。

## 第二阶段：感知接入

手动注入通过后，再接入真实动态障碍来源：

- 深度相机点云检测障碍；
- 已知 actor/model 位姿仅用于诊断；
- 将障碍包围盒发布为 `CollisionObject`；
- PICK、PLACE、拍照位移动、收臂前刷新 PlanningScene。

## 机械臂响应延时口径

当前 G5 机械臂避障响应延时定义为：

```text
total_response_latency = t_plan_result - t_obstacle_trigger
```

其中：

- `t_obstacle_trigger`：G5 脚本决定注入动态障碍盒的墙钟时刻；
- `t_scene_applied`：`/apply_planning_scene` 成功返回的墙钟时刻；
- `t_plan_result`：`/plan_kinematic_path` 返回规划结果的墙钟时刻。

拆分统计：

```text
apply_scene_latency = t_scene_applied - t_obstacle_trigger
planning_latency = t_plan_result - t_scene_applied
total_response_latency = t_plan_result - t_obstacle_trigger
```

当前还没有接入真实视觉，所以 `t_obstacle_trigger` 暂由脚本触发代表。后续接入深度相机后，
`t_obstacle_trigger` 应替换为深度相机/点云首次观测到障碍的 ROS 时间戳。

如果要测“避让动作真正开始”，则终点可改为：

- MoveIt 新规划请求发出；
- 新轨迹被发送到 `joint_trajectory_controller`；
- 关节轨迹首个点明显不同于无障碍基线。

第一版不建议直接用最终轨迹变化作 G5 指标，因为它混入了规划器耗时、控制器排队和仿真慢速因素。

## 数据产出

`benchmarks/g5_arm_avoidance_demo.py` 每次运行都会在 `benchmarks/results/` 下生成：

- `g5_arm_avoidance_<时间戳>.csv`
- `g5_arm_avoidance_<时间戳>.md`

关键字段：

- `case_label`：测试场景标签；
- `expected_verdict`：该场景期望结果，可为 `PLAN_OK`、`PLAN_FAILED` 或 `ANY`；
- `expected_met`：实际结果是否满足场景预期；
- `verdict`：`PLAN_OK` / `PLAN_FAILED` / `EXECUTE_OK` / `EXECUTE_FAILED`
- `apply_scene_latency_ms`：障碍盒注入 PlanningScene 的耗时
- `apply_scene_service_wait_ms`：本次脚本等待 `/apply_planning_scene` 服务可用的耗时
- `apply_scene_call_latency_ms`：服务已可用后，实际 apply PlanningScene 的调用耗时
- `planning_latency_ms`：MoveIt 规划耗时
- `total_latency_ms`：从障碍触发到规划结果返回的总响应延时
- `control_latency_ms`：扣除服务发现等待后的控制响应延时，近似对应移植到常驻节点后的响应
- `moveit_error_code`：MoveIt 规划错误码
- `trajectory_points`：返回轨迹点数
- `obstacle_center` / `obstacle_size`：障碍盒位置和尺寸
- `target` / `start_joints`：目标 TCP 位姿和起始关节位形

## 响应延时怎么看

单次报告重点看三行：

```text
PlanningScene 注入: ok=True, latency=29.4 ms
MoveIt 规划: ok=True, error_code=1, trajectory_points=26, latency=40.6 ms
总响应延时: 140.3 ms
```

含义：

- `apply_scene_latency_ms`：动态障碍盒被注入 MoveIt PlanningScene 的耗时；
- `planning_latency_ms`：MoveIt 在障碍已经生效后返回规划结果的耗时；
- `total_latency_ms`：从障碍触发到规划结果返回的总响应延时。

当前 G5 口径下，对照“动态避障响应延时 <=200ms”时，优先看：

```text
total_latency_ms <= 200
```

例如：

- 可绕开小障碍样例 `total_latency_ms=140.3ms`，该轮满足 `<=200ms`；
- 阻断障碍样例 `total_latency_ms=3894.4ms`，该轮不满足 `<=200ms`，但如果该场景的
  `expected_verdict=PLAN_FAILED`，它仍表示“安全拒绝不可行规划”，不代表 G5 机制失败。

多轮统计时重点看：

- `总响应延时 P95`；
- `总响应延时 >200ms 样本数`；
- `总响应延时 <=200ms 达标率`；
- `服务已连接后的控制响应延时 P95`；
- `控制响应延时 <=200ms 达标率`；
- `预期满足率`。

一句话：单次先看 `total_latency_ms`，如果脚本每轮都重新启动，还要看
`control_latency_ms`，避免把服务发现等待误算成机械臂避障响应。

正式报告建议每类场景至少 20 次，汇总均值、标准差、P95、最大值和 `PLAN_OK` 比例。
注意：`PLAN_FAILED` 不天然表示 G5 失败。如果场景本身是“阻断障碍/不可行目标”，
`PLAN_FAILED` 是安全拒绝规划，应该把 `expected_verdict` 设为 `PLAN_FAILED`，并按
`expected_met=True` 计入预期满足。

多轮汇总：

```bash
python3 benchmarks/g5_arm_avoidance_summary.py benchmarks/results
```

该命令读取 `g5_arm_avoidance_*.csv`，生成：

- `g5_arm_avoidance_summary_<时间戳>.md`

汇总字段包括：

- 样本数；
- `PLAN_OK` / `PLAN_FAILED` 数量；
- 规划成功率；
- 有预期场景数、预期满足数、预期满足率；
- PlanningScene 注入延时 mean/std/P95/max；
- MoveIt 规划延时 mean/std/P95/max；
- 总响应延时 mean/std/P95/max；
- 总响应延时超过 200ms 的样本数。
- 总响应延时 `<=200ms` 达标率。
- 服务已连接后的控制响应延时 mean/std/P95/max；
- 控制响应延时 `<=200ms` 达标率。
