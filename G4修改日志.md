## 使用方法

默认抓取运行不变。采集接触力时，以独立、短时的采集模式启动：

```bash
ros2 launch lab_cobot_gazebo world.launch.py \
  gui:=false enable_contact_force:=true \
  require_finger_contact:=true
```

另开终端启动记录器，再执行一次抓取：

```bash
ros2 run lab_cobot_manipulation contact_force_recorder \
  --duration 240 --output-dir g4_artifacts \
  --stem g4_contact_force_run_2026-07-20
```

输出为 `g4_artifacts/<stem>.csv` 和 `g4_artifacts/<stem>.png`。PNG 可直接作为报告的“接近→接触→闭合”曲线素材，CSV 保留所有原始点用于统计复核。

## 诚实边界

`enable_contact_force:=true` 会把探针从“仅接触判定”切换为真实碰撞响应。项目此前为了避免双探针接触响应把轻质样件推出抓取封套，默认关闭该响应；本轮采集模式也复现了这一物理不稳定性。因此：

- 默认任务和完整仿真必须保持 `enable_contact_force:=false`；
- 力曲线模式仅用于短时采集，不作为抓取成功率验证配置；
- 人为手动 detach 的脱落探针按本次需求未执行；持有监控的运行时检测、失败报告和规划场景清理已完成。

## 轨迹执行时间过长定位与快速专项模式（2026-07-22 11:23–11:38）

本轮先单独解决“移动太慢/轨迹执行时间长”的问题，未把完整 A→B 成功作为本小节判定
目标。

### 现象与原因

首条 A 点 approach 轨迹并不是控制器卡死：`joint_trajectory_controller` 能接收 goal，
最终也能完成；但墙钟耗时异常长。

对照结果：

```text
旧配置：APPROACH_OK=True
旧配置：APPROACH_WALL_SEC=112.712
```

进一步观察到两个原因叠加：

1. MoveIt/IK 曾选择 shoulder/wrist 多圈等价解，关节会走接近 ±2π 的长路径；
2. 抓取专项固定 A/B 坐标运行时，URDF 仍无条件启用 bench RGB-D、LiDAR、IMU：
   - `/bench_camera/*` 深度相机 640x480 @ 15Hz；
   - `/scan` LiDAR；
   - `/imu/data` IMU；
   这些传感器对固定坐标抓取专项没有帮助，却显著拖慢 Gazebo Classic/WSL 下的物理和
   控制器服务响应。

因此“慢”的根因不是后台进程抢满 CPU，也不是 trajectory controller 无法执行；主要是
**MoveIt 规划空间过宽 + Gazebo 无关传感器负载过高**。

### 本轮修改文件

- `src/lab_cobot_moveit/config/joint_limits.yaml`
  - 给 shoulder/wrist 关节增加规划层 `[-pi, pi]` 位置限制，避免 MoveIt 选中多圈绕转
    等价解；
  - 给 6 个 UR 关节补充显式速度限制，供时间参数化使用。
- `src/lab_cobot_moveit/test/test_moveit_config.py`
  - 增加 joint limit 回归测试，确保后续不会重新放开多圈路径。
- `src/lab_cobot_description/urdf/lab_cobot.urdf.xacro`
  - 新增 `enable_lab_sensors` xacro 参数；
  - 默认仍为 `true`，整仿真/感知方向不受影响；
  - 设为 `false` 时不生成底盘 LiDAR、IMU 和 bench RGB-D。
- `src/lab_cobot_gazebo/launch/world.launch.py`
  - 新增 launch 参数 `enable_lab_sensors`，并传入 xacro；
  - 抓取专项固定坐标验证可用 `enable_lab_sensors:=false` 降低 Gazebo 负载。

## G4 A→B 完整抓取放置最终修复与验证（2026-07-22 12:00–12:35）

### 目标

完成 G4 抓取方向任务中“在 A 处抓取物品，并在 B 处安全放下”的闭环验证。按用户要求，
“人为制造脱落”暂不处理。

### 本轮失败原因复盘

1. **A/B 高度坐标系用错**
   - `PickPlace.pick()` / `PickPlace.place()` 接收的是 `base_link` 坐标；
   - 早期脚本直接传入世界坐标/台面高度，导致 TCP 目标整体偏高，夹爪在物块上方，触觉
     接触不稳定；
   - TF 实测 `base_footprint <- base_link` 的 z 偏移为 `0.155 m`，因此专项脚本改用：
     - A：`[0.80, 0.00, 0.630]`
     - B：`[0.82, 0.20, 0.595]`

2. **触觉抓取接近高度过高**
   - 原触觉 approach 高度让下降段较长且容易错过稳定双指接触；
   - 已将触觉 approach 高度调低，并在 approach/lift/transfer 阶段临时使用较快机械臂
     速度比例，避免再次出现“轨迹没坏但墙钟超长”的现象。

3. **插件双指接触判定过严**
   - Gazebo 接触消息可能左右手指不在同一 physics tick 出现；
   - 原逻辑要求同一帧同时满足双指接触，导致实际已经碰到但 attach 被拒绝；
   - 已改为保存左右手指最近接触时间，在 `0.2 s` 仿真时间新鲜窗口内认为双指接触成立。

4. **抓取封套 y 方向过窄**
   - 真实接触/夹取过程中物块有轻微横向偏差；
   - `max_abs_y` 从 `0.070` 放宽到 `0.105`，仍保留 x/z/总距离约束，避免误抓远处物体。

5. **持有状态心跳对 WSL/Gazebo 低实时因子过敏**
   - 转运过程中 Gazebo 实时因子低，`hold_status` 心跳在墙钟上可能很久才刷新；
   - 原 `0.5 s` 墙钟新鲜度会把实际仍 attached 的状态误判为丢失；
   - 已改为 `300.0 s`，避免专项仿真长轨迹期间误报丢失。

6. **触觉放置释放点过低**
   - 原触觉放置使用 `target_z + 0.025 - 0.05`，等于把 TCP 放到目标点下方；
   - B 点下降段因此出现 MoveIt `Unable to sample any valid states for goal tree` /
     `descent move failed`；
   - 已取消该额外下压补偿，使触觉放置在 `target_z + 0.025` 悬空释放。

7. **Gazebo fixed joint 持物不稳定**
   - 插件日志显示 `attached`，但物块没有稳定随腕部运动到 B；
   - 原因是 Gazebo Classic 中跨模型 fixed joint 在本项目的位姿驱动/接触设置下不够可靠；
   - 已在插件持有分支增加每 tick 的持有位姿钳制：按抓取瞬间的腕部相对位姿同步物块，
     release 时物块就在当前腕部/B 点附近释放。

## 整个 G4 运行流程说明（稳定 A→B + 持有监控 + 力曲线）

### 总体原则

G4 当前拆成两类运行模式：

1. **稳定 A→B 验证模式**
   - 目标：证明机器人能在 A 处抓取物块，并在 B 处安全放下；
   - 使用 `enable_contact_force:=false`；
   - 启用触觉接触判定、attach/release、持有监控；
   - 不采集真实接触力曲线；
   - 这是“最终能否跑通”的主要验收模式。

2. **力曲线专项采集模式**
   - 目标：生成左右探针力值 CSV/PNG；
   - 使用 `enable_contact_force:=true`；
   - 会开启触觉探针真实接触响应；
   - 该模式可能把轻质样件推偏，稳定性低于 A→B 验证模式；
   - 因此力曲线采集与稳定放置验证分开做，不把力曲线采集模式作为最终 A→B 稳定验收。

### 持有监控在流程中的位置

持有监控不是单独运行的脚本，而是嵌入在稳定 A→B 流程中：

1. Gazebo grasp 插件 attach 后发布：

3. `PickPlace` 在 pick 成功、物块进入搬运阶段后启动持有监控；
4. approach、transfer、place descent 期间持续检查物块是否仍被插件确认持有；
5. 到达释放位后，先停止持有监控，再请求 release；
6. release 阶段 `hold_status` 从 `holding` 切换到 `released` 属于正常放下，不再被误报为“搬运中丢失”。

如果搬运中持有监控失败，日志会出现：

```text
持有监控失败：抓取插件未确认物块仍附着，终止后续抓放步骤
抓放任务失败：物块在搬运期间不再被持有
```

这属于失败，应排查 attach、fixed joint、hold_status 心跳或 Gazebo 实时因子。

```bash
import threading
import time

import rclpy
from gazebo_msgs.srv import GetEntityState
from rclpy.executors import MultiThreadedExecutor

from lab_cobot_manipulation.pick_place_node import PickPlace

A = [0.80, 0.00, 0.630]
B = [0.82, 0.20, 0.595]

rclpy.init(args=["--ros-args", "-p", "use_sim_time:=true"])
node = PickPlace(
    target_object="aruco_sample",
    use_tactile_grasp=True,
    use_planning_scene_obstacles=True,
)
executor = MultiThreadedExecutor(num_threads=3)
executor.add_node(node)
threading.Thread(target=executor.spin, daemon=True).start()


def pose(label):
    client = node.create_client(GetEntityState, "/gazebo/get_entity_state")
    if not client.wait_for_service(timeout_sec=10.0):
        print(f"{label}_ARUCO_POSE=None service_timeout", flush=True)
        return None
    req = GetEntityState.Request()
    req.name = "aruco_sample"
    req.reference_frame = "world"
    future = client.call_async(req)
    deadline = time.monotonic() + 10.0
    while not future.done() and time.monotonic() < deadline:
        time.sleep(0.02)
    if not future.done() or not future.result().success:
        print(f"{label}_ARUCO_POSE=None no_result", flush=True)
        return None
    p = future.result().state.pose.position
    q = future.result().state.pose.orientation
    out = (
        round(p.x, 4), round(p.y, 4), round(p.z, 4),
        round(q.x, 4), round(q.y, 4), round(q.z, 4), round(q.w, 4),
    )
    print(f"{label}_ARUCO_POSE={out}", flush=True)
    return out


try:
    print("G4_STABLE_AB_WITH_FORCE_START", flush=True)
    pose("INITIAL")

    t0 = time.monotonic()
    pick_ok = node.pick(A)
    pick_dt = time.monotonic() - t0
    print(f"G4_PICK_OK={pick_ok}", flush=True)
    print(f"G4_PICK_WALL_SEC={pick_dt:.3f}", flush=True)
    pose("POST_PICK")

    place_ok = False
    place_dt = 0.0
    if pick_ok:
        t1 = time.monotonic()
        place_ok = node.place(B)
        place_dt = time.monotonic() - t1

    print(f"G4_PLACE_OK={place_ok}", flush=True)
    print(f"G4_PLACE_WALL_SEC={place_dt:.3f}", flush=True)

    time.sleep(2.0)
    final = pose("FINAL")
    if final:
        x, y, z, *_ = final
        on_b_xy = abs(x - 0.82) <= 0.25 and abs(y - 0.20) <= 0.10
        on_table_z = 0.74 <= z <= 0.83
        print(f"FINAL_ON_B_XY={on_b_xy}", flush=True)
        print(f"FINAL_ON_TABLE_Z={on_table_z}", flush=True)
        print(f"FINAL_ON_B_TABLE={on_b_xy and on_table_z}", flush=True)
        print(
            f"G4_STABLE_AB_WITH_FORCE_OK={bool(pick_ok and place_ok and on_b_xy and on_table_z)}",
            flush=True,
        )
finally:
    executor.shutdown()
    node.destroy_node()
    if rclpy.ok():
        rclpy.shutdown()
PY
```

### 8. 常见注意事项

- 所有终端必须统一 `RMW_IMPLEMENTATION=rmw_fastrtps_cpp`。
- G4 稳定验收不要开 `enable_contact_force:=true`；那是旧的真实接触力专项模式，可能把样块推飞。
- 如果 GUI 很卡，先用 `gui:=false` 跑通结果，再单独开 GUI 看场景。
- 每次完整 A→B 跑完后，如果要重跑，最好重启 Gazebo world，让样块回到 A 点初始位置。
- 退出时 MoveIt/Gazebo 可能打印 `class_loader`、`exit code -11` 或 SIGKILL；只要这些出现在
  `FINAL_ON_B_TABLE=True` 之后，就按退出阶段噪声处理。
